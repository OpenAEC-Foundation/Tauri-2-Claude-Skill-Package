# tauri-impl-build-deploy: Anti-Patterns

These are confirmed error patterns from the Tauri 2 build and deployment system. Each entry documents the WRONG pattern, the CORRECT pattern, and WHY.

Sources: https://v2.tauri.app/distribute/, vooronderzoek-tauri.md Section 6 and Section 10.6

---

## AP-001: Losing the Updater Private Key

**WHY this is wrong**: The updater system uses asymmetric cryptography. Existing installations verify updates using the embedded public key. Without the matching private key, you cannot sign updates that existing installations will accept. There is no recovery mechanism.

ALWAYS back up the private key generated by `tauri signer generate` in a secure location (e.g., a secrets manager, encrypted vault). NEVER store it in version control.

---

## AP-002: Setting createUpdaterArtifacts Without Signing Keys

**WHY this is wrong**: When `createUpdaterArtifacts` is `true`, the build process attempts to sign the update artifacts. Without `TAURI_SIGNING_PRIVATE_KEY` set, the build fails.

```json
// WRONG: build fails if TAURI_SIGNING_PRIVATE_KEY not set
{
  "bundle": {
    "createUpdaterArtifacts": true
  }
}
```

```bash
# CORRECT: set environment variable before building
export TAURI_SIGNING_PRIVATE_KEY="content-or-path-to-key"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD=""
tauri build
```

ALWAYS set `TAURI_SIGNING_PRIVATE_KEY` in the build environment (locally and in CI) before enabling `createUpdaterArtifacts`.

## AP-003: Distributing Unsigned macOS Builds

**WHY this is wrong**: macOS Gatekeeper quarantines unsigned applications. Users see a warning that the app is from an unidentified developer and cannot open it through normal means.

```json
// WRONG: no signing identity
{
  "bundle": {
    "macOS": {}
  }
}
```

```json
// CORRECT: at minimum, use ad-hoc signing for testing
{
  "bundle": {
    "macOS": {
      "signingIdentity": "-"
    }
  }
}

// CORRECT: use Developer ID for distribution
{
  "bundle": {
    "macOS": {
      "signingIdentity": "Developer ID Application: Your Name (TEAMID)"
    }
  }
}
```

ALWAYS sign macOS builds. Use `"-"` for ad-hoc (testing only). Use Developer ID + notarization for production distribution.

## AP-004: Using v1Compatible for New Projects

**WHY this is wrong**: The `"v1Compatible"` value for `createUpdaterArtifacts` generates artifacts in the v1 format. This is only needed when migrating from Tauri v1 where existing installations expect the old format. New projects should use the native v2 format.

```json
// WRONG: unnecessary v1 compatibility for new project
{
  "bundle": {
    "createUpdaterArtifacts": "v1Compatible"
  }
}
```

```json
// CORRECT: use boolean true for new projects
{
  "bundle": {
    "createUpdaterArtifacts": true
  }
}
```

ALWAYS use `true` for new projects. ONLY use `"v1Compatible"` when migrating from Tauri v1.

## AP-005: Not Committing Cargo.lock

**WHY this is wrong**: Without `Cargo.lock`, dependency versions can differ between builds, causing non-deterministic build outputs and potential breakage.

```gitignore
# WRONG: ignoring Cargo.lock
src-tauri/Cargo.lock
```

```gitignore
# CORRECT: only ignore target directory
src-tauri/target/
```

ALWAYS commit `src-tauri/Cargo.lock` to version control.

---

## AP-006: Missing beforeBuildCommand

**WHY this is wrong**: Without `beforeBuildCommand`, the frontend assets are not built before Tauri bundles the application. The resulting installer will contain stale or missing frontend files.

```json
// WRONG: no build command configured
{
  "build": {
    "frontendDist": "../dist"
  }
}
```

```json
// CORRECT: frontend is built automatically
{
  "build": {
    "frontendDist": "../dist",
    "beforeBuildCommand": "npm run build"
  }
}
```

ALWAYS set `beforeBuildCommand` to your frontend build script.

## AP-007: Missing Linux Dependencies in CI

**WHY this is wrong**: Linux builds require system libraries that are not pre-installed on CI runners. The build fails with compilation errors about missing headers.

```yaml
# WRONG: no Linux dependencies installed
- name: Build Tauri app
  uses: tauri-apps/tauri-action@v0
```

```yaml
# CORRECT: install dependencies first
- name: Install Linux dependencies
  if: runner.os == 'Linux'
  run: |
    sudo apt-get update
    sudo apt-get install -y libwebkit2gtk-4.1-dev libappindicator3-dev librsvg2-dev

- name: Build Tauri app
  uses: tauri-apps/tauri-action@v0
```

ALWAYS install `libwebkit2gtk-4.1-dev`, `libappindicator3-dev`, and `librsvg2-dev` on Linux CI runners.

## AP-008: GitHub Token Without Write Permissions

**WHY this is wrong**: `tauri-action` creates GitHub releases and uploads artifacts. Without write permissions, the action fails silently or with a 403 error.

```yaml
# WRONG: no permissions block
jobs:
  release:
    runs-on: ubuntu-latest
```

```yaml
# CORRECT: explicit write permission
permissions:
  contents: write

jobs:
  release:
    runs-on: ubuntu-latest
```

ALWAYS set `permissions: contents: write` at the workflow or job level when using `tauri-action` for releases.

---

## AP-009: Not Implementing Update UI in v2

**WHY this is wrong**: Tauri v2 removed the automatic update dialog that existed in v1. Without explicit update checking code, users will never receive updates even if the updater is configured.

```rust
// WRONG: plugin registered but never used
app.handle().plugin(tauri_plugin_updater::Builder::new().build());
// No check() call anywhere
```

```rust
// CORRECT: actively check for updates
app.handle().plugin(tauri_plugin_updater::Builder::new().build());

let handle = app.handle().clone();
tauri::async_runtime::spawn(async move {
    if let Some(update) = handle.updater().unwrap().check().await.unwrap() {
        // Download, install, and restart
        update.download_and_install(|_, _| {}, || {}).await.unwrap();
        handle.restart();
    }
});
```

ALWAYS implement explicit update checking logic (in Rust or JavaScript) when using the updater plugin.
